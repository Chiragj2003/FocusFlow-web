API behavior requirements
- All API routes require Clerk auth, except public webhooks if added.
- Use JSON responses with consistent shape: { data, error }.
- Validate inputs and return 400 for bad input, 401 for unauthenticated, 404 for not found, 500 for server errors.
- Support date range query params for entries, insights, and exports.
- Normalize timezone handling using user timezone from settings.

Design patterns to mention
- App Router with route groups to separate public and protected areas.
- Server components handle data fetching, serialization, and auth checks; client components handle interactivity.
- A lib layer for domain logic (analytics, badges, templates, challenges) to keep UI thin.
- API route handlers enforce auth and validation, returning consistent JSON responses.
- Centralized Prisma client singleton to avoid connection storms in dev.
- Incremental revalidation and caching to balance freshness and performance.
- Clear separation of UI state: loading, empty, and error states are explicit on each page.

Basic workflow to explain
1) Public landing and auth with Clerk, redirecting into the protected app on success.
2) On first app load, upsert user record with email, name, and default timezone.
3) User creates habits via /api/habits; habits appear in the monthly grid and list.
4) User logs entries via quick toggle or value input; /api/entries persists data.
5) Dashboard aggregates metrics and charts from analytics helpers.
6) Insights compute streaks, heatmap values, and top habit rankings.
7) Badge checks run after key actions and award new badges when thresholds are met.
8) Challenges can be joined; progress updates based on daily completion.
9) Exports generate CSV/PDF on demand with date ranges and summaries.
10) Settings updates timezone and preferences, affecting analytics and reminders.
11) PWA runs as a responsive app with installable manifest and service worker.
